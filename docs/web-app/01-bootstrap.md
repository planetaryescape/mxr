# Phase 1 — Bootstrap

Goal: an empty SPA that the user can launch via `mxr web`, that authenticates against the daemon bridge, that shows a working app shell, and that proves the WebSocket event stream end-to-end. After Phase 1, the SPA does basically nothing yet — but the infrastructure is solid.

## Deliverables

1. `apps/web/` skeleton with all tooling (Vite, TS strict, Tailwind 4, oxlint/oxfmt, Vitest, Playwright config).
2. Generated API types (`src/api/generated.ts`) from the bridge OpenAPI spec.
3. Typed API client (`openapi-fetch` wrapper) with auth middleware.
4. Reconnecting WebSocket client.
5. Token bootstrap: read URL fragment → localStorage → fallback paste panel.
6. TanStack Query client + provider.
7. Zustand stores stubbed (selection, modals, connection, UI prefs).
8. AppShell with Sidebar, Topbar, StatusBar, RightRail (all initially stubs).
9. TanStack Router with placeholder routes for every URL in the IA table.
10. Sonner toaster mounted at root.
11. Dev page (`/dev`) with a smoke check that hits `/api/v1/admin/status` and displays the result.
12. **Rust changes**:
    - Extend `Command::Web` with `--no-open`, `--remote-host`, optional `--bridge-port` already exists via `--port`.
    - Rewrite `crates/daemon/src/commands/web.rs` to persist token, open browser, handle remote-host (no local bind).
    - Add `bridge_token_path()` to `crates/config/src/lib.rs`.
    - Add `web-ui` cargo feature to `crates/web` that pulls in `include_dir`.
    - In `crates/web/src/lib.rs`, add SPA serving at `/` (and any non-`/api` path → SPA index.html) when `web-ui` feature is enabled, with strict CSP.
    - When `MXR_WEB_DEV_PROXY=http://localhost:5173` env var is set, the bridge proxies non-`/api` requests to the Vite dev server instead of serving embedded assets.
    - Update `crates/daemon/tests/snapshots/cli_help__cli_help_web.snap` (or generate one if it doesn't exist).

## Files to create / modify — Rust

- `crates/config/src/lib.rs` — add:

  ```rust
  pub fn bridge_token_path() -> std::path::PathBuf {
      config_dir().join("bridge-token")
  }
  ```
  Plus a `read_or_create_bridge_token() -> std::io::Result<String>` helper that:
    1. Reads existing file if present.
    2. Otherwise generates a new UUID, writes with mode 0600, returns it.
- `crates/daemon/src/cli/mod.rs` — extend `Command::Web` to:

  ```rust
      #[arg(long, default_value = "127.0.0.1")]
  ```

## API endpoints used

- `GET /api/v1/admin/health` (unauthenticated, version probe)
- `GET /api/v1/admin/status` (smoke check on `/dev`)

## State management notes

- **`uiPrefsStore`**: theme, density, sidebar-collapsed. Persisted to localStorage (`mxr.uiPrefs`). Hydrated synchronously in `main.tsx` before render so theme doesn't flash.
- **`connectionStore`**: WS state (`connecting | connected | reconnecting | offline`), last-event timestamp, bridge URL.
- **`tokenStorage`**: tiny module — `getToken()`, `setToken(t)`, `clearToken()`, `bootstrapFromHash()` (reads `#token=...` then `history.replaceState({}, "", location.pathname + location.search)`).

## Auth bootstrap pseudocode

```ts
// main.tsx (before any provider mounts)
import { bootstrapFromHash, getToken } from "@/lib/tokenStorage";

bootstrapFromHash();
const token = getToken();
// if (!token) → still render the app; api client will surface 401 → settings/token panel.
```

## TanStack Router setup notes

- Plugin already configured in `vite.config.ts`. It generates `src/routeTree.gen.ts` automatically.
- Each route file exports `Route` via `createFileRoute(path)`. Use `Route.useSearch()` and `Route.useParams()` for typed access.
- `__root.tsx` mounts `<AppShell />` and renders `<Outlet />`. Devtools (`@tanstack/react-router-devtools`) only in dev.

## TanStack Query setup notes

- Single client with sensible defaults: `staleTime: 30_000`, `refetchOnWindowFocus: false` (we have WS).
- DevTools in dev only.
- Query key factory pattern (object form) so `invalidateQueries({ queryKey: ["envelopes"] })` clears every variant.

## WebSocket client notes

- Connects to `/api/v1/events` using `Sec-WebSocket-Protocol: bearer, <token>` (bridge already accepts this).
- Reconnect with exponential backoff (250ms → 8s, jittered), 25s heartbeat ping, online/offline + window-focus reconnect triggers.
- Exposes `subscribe(handler)` and a Zustand-backed `connectionStore` for status indicators.
- Mount once via `<DaemonEventsProvider>` inside `__root.tsx` (children of QueryClientProvider).

## Common pitfalls

- The bridge already serves Swagger UI at `/api/v1/docs` and OpenAPI JSON at `/api/v1/openapi.json` — make sure SPA fallback routes don't shadow these.
- The bridge auto-rejects non-loopback binds without TLS (see `middleware.rs`). For `--remote-host`, the local CLI **does not bind** anything; it just opens the browser to the remote URL. The remote operator handles TLS.
- TSR's auto-codegen plugin watches `routes/` — saving any route file regenerates `routeTree.gen.ts`. Do **not** edit that file by hand.
- shadcn components depend on `lib/utils.ts:cn()`. Create that early.
